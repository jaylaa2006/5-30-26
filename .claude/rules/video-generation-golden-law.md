# Video Generation — Golden Law (site-wide standard)

**Status:** PROJECT LAW. Established 2026-05-21 by user directive: the
intro-video generation algorithm is the single standard for ALL video
across the site, going forward — to lock site-wide visual consistency.
Auto-loads via CLAUDE.md. Supersedes the per-surface "body-holds" motion
grammar for ritual/reader/foundation content.

## HARD RULE 0 (MUST — no exceptions) — video prompts come ONLY from the canonical generator

**NEVER hand-author, hand-edit, hand-improvise, or LLM-write a Veo/Flow video
prompt.** Locked by user directive 2026-05-30. EVERY video prompt for the site
MUST be produced by the canonical in-repo generator for its category. The
generator owns the chunk-sourced distillation, the `LEAN_SAFETY` + `scrub()` /
`ULTRA_SAFE` RAI handling, the mandatory Africana representation block, and the
`briefToVeoPrompt` golden-law grammar. Hand-rolling a prompt drifts the
site-wide look AND silently skips the RAI scrub — both forbidden.

| Category | Canonical generator |
|---|---|
| Story intros / outros (incl. pert-em-heru, sage-peh, Yeshua's Way, general library) | `generate-story-intros-v2.mjs --story <id> --mode intro\|outro` |
| Senebty ritual clips | `generate-senebty-veos.mjs` |
| Set bookends | `generate-set-videos.mjs` |
| Hero portraits (Imagen — portraits ONLY) | `generate-hero-portraits.mjs` |

- **To produce a PDF / text list of prompts for MANUAL Flow generation**, run the
  generator with **`--dry-run`** and capture its emitted prompt verbatim. The
  `--dry-run` path distills + scrubs but does NOT spend Veo credit. This rule
  applies *even when "just making a PDF of prompts"* — that was the exact
  loophole that prompted this rule.
- If a story's data is not reachable by a generator's parser, FIX THE PARSER /
  point it at the real source — do not fall back to hand-writing.
- The ONLY hand-editing permitted is on a generator-emitted prompt to apply a
  documented RAI substitution from the catalog, and even then prefer the
  generator's `--mode`/ultra-safe retry.

## The law

1. **ONE PIPELINE FOR ALL MOTION VIDEO — substrate → enhancement.** Every
   video on the site — story intros, outros, set bookends, foundation ritual
   clips, ambient backdrops — is generated by the intro-video algorithm:
   **Veo 3.1 text-to-video, cel-shaded Spider-Verse style, full cinematic
   motion**. This is the **SUBSTRATE**. Every specialized video STARTS from
   the substrate and adds an **enhancement layer** of relevant context
   (battle → battlefield-dawn tension; foundation ritual → the practice in
   motion; set bookend → the journey frame) — never a separate base.
   - **Per-story scene context is DERIVED FROM THE STORY'S OWN CHUNKS**, not
     from generic scene-type templates. Read the actual `chunk-0` text +
     title + principle + character descriptions from `maat-reader.html`
     `const STORIES = [...]`, distill into a clean establishing-shot scene
     brief, then generate. This is the SAME pattern `generate-art-v2.js`
     already uses for chunk art ("reads the ACTUAL story text … EXACT chunk
     text … consistent character descriptions") — reuse its story-parser so
     intros and art stay consistent. Generic `scene-temple`/`scene-nile`
     templates in the legacy `generate-intro-videos.mjs` are the OLD way and
     are being replaced by chunk-sourced context.
   - The grade-2 intros are the gold substrate the whole site matches.

2. **FULL MOTION IS MANDATORY; "BODY-HOLDS" IS RETIRED.** Any clip that
   depicts an action (a ritual, a journey, a gesture) MUST show the action
   with full character + camera movement (the F1 reference: Sitra walks,
   then drinks). The old "body-holds / camera-locked / one-element-moves"
   constraint is retired for all ritual/reader/foundation content. Rule:
   natural motion always; intensity scaled to the moment — full for
   establishing + ritual-demonstration, **gentle but LIVING** for blessings
   (a nod, a breath, an ankh-lift). Never a frozen figure. Realistic-static
   "dead" video is the failure mode this law exists to kill.

3. **IMAGEN 4 ULTRA IS RESERVED STRICTLY FOR HERO PORTRAITS** of the greats
   of African history (static portrait cards in `art/heroes/`, `art/rulers/`,
   and the foundation index-card portraits). It must NEVER be used for reader
   chunk art, foundation content, or any animated/motion surface. The v3.48
   "elite pipeline" that anchored foundation chunk art to realistic Imagen
   portraits is retired for motion content — it produced the "dead" look.
   Realistic stills do not animate.

4. **AFRICANA REPRESENTATION DISCIPLINE** (non-negotiable, every prompt):
   dark-skinned African / Nubian-Nilotic features, the anti-lightening block,
   no Europeanization (per ben-Jochannan / Clarke / Diop / Obenga / Van
   Sertima). Aramaic name fidelity where applicable (Yeshua's Way). See
   CLAUDE.md Critical Rules + `skills/docs/art/`.

5. **ENGINE = VEO 3.1 VIA VERTEX AI** on the GCP project (model
   `veo-3.1-generate-001`, 1080p), run through the repo's generation scripts
   — automated, batch, reproducible, in-repo. ADC auth (`gcloud auth
   application-default login`). Google Flow / consumer AI subscriptions are a
   MANUAL FALLBACK only (e.g. for longer stitched loops), never the standard
   batch path. Vertex returns the clip as inline base64 (`videoBytes`) when
   no `outputGcsUri` is set — write those bytes directly; the Gemini-API
   `files.download()` path does NOT apply in Vertex mode.

6. **READ-AND-WAIT SURFACES GET LONGER LOOPS.** Where a video loops while the
   user reads and waits to advance (foundation reader backdrops), prefer
   longer clips to reduce visible repetition. Plain ~8s is fine for one-shot
   intros.

7. **PERFORMANCE + CACHE DISCIPLINE STILL APPLY.** Enterprise Rule 5
   (cache-buster `?v=` bumps on changed assets) and Rule 7 (backdrop video
   budget: viewport-gated, poster-backed, bitrate-budgeted, mobile-encoded
   via `scripts/encode-*.sh`). Generation quality does not waive delivery
   discipline.

## Vertex Veo RAI constraints (learned 2026-05-21, carpenter-of-nazareth debug)

Vertex Veo (`veo-3.1-generate-001`) rejects prompts at SUBMIT time (error
code 3, "contains words that violate usage guidelines") — these blocks are
FREE (no generation billed), so iterate on them cheaply. Confirmed triggers:

1. **Depicting an identifiable real religious/public figure.** "Yeshua/Jesus
   of Nazareth" as the subject blocks. NOTE: the name *alone* ("a man named
   Yeshua") passes — it's the IDENTIFYING COMBINATION (name + place + role)
   that trips it. Fix: describe the protagonist GENERICALLY by role ("a young
   carpenter", "a queen", "a scribe") + evoke the WORLD. This will also hit
   general-library intros naming real pharaohs/kings (Hatshepsut, Mansa Musa,
   Taharqa, Imhotep) — genericize them too.
2. **Sensitive tokens inside NEGATIVES.** "NO Latin crosses, no halos, no
   Western-Christian iconography" in a negative clause BLOCKS — the filter
   reads the tokens regardless of the "NO". State direction POSITIVELY
   (dark-skinned people of the Kemet continuum), never enumerate forbidden
   religious/violent symbols. (Mirrors the senebty audio-polarity RAI lesson
   in feedback_moving_art_grammar.)
3. **Weapon-adjacent tool inventories** (adze/awl/drill-bow/blade) and
   over-dense prompts are riskier — keep briefs lean, soften tools to "a
   woodworking tool".

Naming real PEOPLE (e.g. citing scholars ben-Jochannan/Diop in the prompt)
is risky on Vertex (looser on the AI Studio API — where the original 12 YW
intros were made). Keep scholar attribution in OUR docs, not in the image
prompt. `generate-story-intros-v2.mjs` encodes all of this (LEAN_SAFETY +
scrub()).

To depict a real religious figure literally, the workarounds are
image-to-video from existing art (figure from the reference image, not the
text) or the AI Studio API / Flow — NOT pure text-to-video on Vertex.

## Character identity (verified reference image)

Pure text-to-video DRIFTS a character's identity — the same prompt yields a
different-looking person each clip (the "who's Yeshua?" wobble the user flagged
2026-05-21). When a recurring character must stay consistent across clips
(Yeshua across YW; Sitra / Merytamun / the foundation cast across foundations):

- Condition on that character's CANONICAL reference image (image-to-video /
  first-frame) — proven to pass Vertex RAI in the B experiment (the figure
  comes from the IMAGE, not the named text).
- Keep ONE canonical reference per recurring character (`art/heroes/` or a
  locked chunk-art frame), and **VERIFY the reference you pull is the targeted
  person before generating** — never pull the wrong character.
- Most critical for the FOUNDATIONS (named recurring cast). Establishing-shot
  story intros with no recurring named character don't need it (their figure
  is generic-by-design).

## Why

Site-wide visual consistency. The grade-2 intro videos are the product's
visual identity — alive, cel-shaded, full-motion. Realistic Imagen stills and
body-held near-static Veos read as "dead" against them and break the spell,
worst on the foundation reader. One pipeline = one look.

## Supersedes

- The "body-holds, elements move" default for ritual/reader/foundation
  content (`feedback_moving_art_grammar`, `feedback_veo_category_budget`).
  Body-holds survives ONLY as "gentle/living, never frozen" for blessing /
  ambient closers — and even those should breathe.
- The v3.48 elite-pipeline realistic foundation chunk art (retired for motion
  content; Imagen reserved for hero portraits only).
